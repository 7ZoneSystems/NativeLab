Draw connections by hand
Drag from any port dot on a block to any port on another. Connections snap as smooth Bézier curves. Delete with a click. It feels like a whiteboard, not a settings panel.
Loops that repeat automatically
Connect a block back to an earlier one and NativeLab detects the loop. Set how many times it repeats — the chain runs that many passes, accumulating context each time. Shown as a cyan ↻×N arrow.
Every stage streams live
As the pipeline runs, each Intermediate block gets its own live tab in the right panel. You see every model's tokens arriving in real time. No black boxes, no waiting for a final dump.
Save and reload any pipeline
Name a pipeline and save it. All blocks, connections, and model assignments are stored as a small JSON file on your machine. Reload with one click — everything is exactly as you left it.
Run a pipeline from chat
Hit the 🔗 Pipeline button next to Send. Pick a saved pipeline and it runs on your current message without leaving chat. Each stage appears as its own labelled bubble in the conversation.
Validation before run
The canvas refuses to run an empty pipeline, catches direct MODEL → MODEL connections, and warns about unreachable blocks before anything starts.

The block catalogue

20+ block types,
five categories.

Pipelines are built from blocks you drag onto the canvas. Each block has a clear job — and the LLM-logic blocks are where NativeLab pulls ahead of every other local-LLM tool.

I/O

Input · Output · Model · Intermediate

The four core blocks. Every pipeline starts with an Input and ends with an Output. Model blocks run any loaded engine. Intermediate blocks stream their own live output tab.

live streaming required
Context

Reference · Knowledge · PDF

Inject documents into the prompt before it hits the model. PDF block auto-summarises large files, Reference uses your session library, Knowledge is a free-form text panel.

RAG auto-summarise
Logic — Python

IF/ELSE · SWITCH · FILTER · TRANSFORM

Sandboxed Python expressions route the pipeline. IF returns a boolean, SWITCH returns a key, FILTER gates the flow, TRANSFORM rewrites context (prefix, suffix, replace, upper, lower, strip, truncate).

no imports no fs/network
Logic — Flow

MERGE · SPLIT · Custom Code

MERGE joins multiple incoming connections (concat, prepend, append, JSON array). SPLIT broadcasts identical context to all outgoing edges. Custom Code is a sandboxed Python editor with a live syntax check and a test runner.

multi-input multi-output
LLM Logic

LLM-IF · LLM-SWITCH · LLM-FILTER

The model itself decides where the pipeline goes. LLM-IF asks a YES/NO question and routes accordingly. LLM-SWITCH classifies into named arms. LLM-FILTER lets the model gate the flow with a structured explanation.

★ Exclusive agentic
LLM Logic

LLM-TRANSFORM · LLM-SCORE

LLM-TRANSFORM has the model rewrite context with an output-only system prompt. LLM-SCORE rates 1–10 and routes to LOW / MID / HIGH arms — the score arm receives the raw numeric value downstream.

★ Exclusive numeric routing

What you build with it

A few example pipelines

Three patterns that show why the visual builder beats hand-rolled scripts. Each is a few minutes of dragging blocks onto the canvas — no config files, no glue code.

Reasoning → Code
A reasoning model thinks through architecture first, then a coding model writes the implementation. Reasoning trace stays visible in the chat — nothing hidden.
INPUT → REASONING MODEL → INTER → CODING MODEL → OUTPUT
Multi-PDF synthesis
Read a folder of PDFs, summarise each one, then synthesise themes across all of them. Pause/resume mid-job — long documents don't have to finish in one session.
INPUT → PDF (each file) → SUMMARIZE → MERGE → CONSOLIDATE → OUTPUT
Agentic LLM router
An LLM-SWITCH classifies the user's intent, then routes to a different downstream model — fast model for chat, reasoning model for analysis, coding model for code. All visual, no Python required.
INPUT → LLM-SWITCH → CHAT / REASON / CODE → OUTPUT

Why this beats hand-rolled scripts

Pipelines you can read,
without reading code.

A canvas of blocks shows what's happening at a glance. A Python script doesn't. That difference is why pipelines ship — and why they get reused across teammates and projects.

01 — VISIBILITY

The pipeline is the diagram

Every stage is on screen. A new teammate understands what a pipeline does in 10 seconds — no reading code, no chasing function calls. Saved pipelines are JSON, but the canvas is the source of truth.

02 — DEBUGGABILITY

Every stage streams

Each Intermediate block gets its own live tab. When something looks wrong, you see exactly which stage produced it — and you see it as the tokens arrive, not after the whole pipeline finishes.

03 — REUSABILITY

Save once, run anywhere

Saved pipelines are small JSON files. Drop one in ~/.nativelab/pipelines/ and it appears in every chat session. Run from chat without leaving the conversation.

04 — COMPOSITION

Local + cloud, in one flow

Use a fast cloud model for the broad strokes and a private local model for the sensitive parts — all in the same pipeline. The API block looks identical to the Model block, so cloud and local compose freely.

05 — AGENTIC

The model can route the flow

LLM-IF, LLM-SWITCH, LLM-FILTER, LLM-SCORE put the model itself in the routing path. You build agentic behaviour visually, without writing prompt-orchestration code.

06 — UNIQUE

No equivalent in the field

LM Studio has a single chat surface. Jan does API multiplexing. AnythingLLM does RAG. None of them have a canvas-based pipeline editor with LLM-logic blocks. See the comparison →
